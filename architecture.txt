Jak bych to udělal já
Nejčistší architektura

Tvoje extension:

přidá command třeba uxp.attach
případně DebugConfigurationProvider jen pro generování návrhů configu
udělá UXP target discovery
vyřeší path mapping / shim endpoint
spustí vestavěný JS debugger přes vscode.debug.startDebugging(...)

Tedy například mentálně takto:

user spustí „Attach to Adobe UXP“
tvoje extension najde správný websocket / shim URL
sestaví config pro JS debugger
zavolá VS Code API pro start session

Tohle:

nevyžaduje speciální práva,
netluče se s built-in debuggerem,
a nevyžaduje, aby js-debug mělo veřejné extension API.
Kdy bys potřeboval vlastní debug adapter

Jen pokud zjistíš, že UXP je natolik odlišné, že js-debug nejde rozumně použít přes běžnou konfiguraci nebo lehký shim. Pak bys měl vlastní type: "uxp" a svůj adapter přes DebugAdapterDescriptorFactory. VS Code tohle explicitně podporuje jako standardní cestu.

Moje doporučení jako finální rozhodnutí

Pro UXP bych šel touto cestou:

1. Vlastní extension ano.
2. Vlastní full debugger ne, dokud to není nutné.
3. Nepoužívat interní API ms-vscode.js-debug.
4. Použít veřejné VS Code debug API a delegovat na vestavěný JS debugger.

To je nejstabilnější varianta.

Můžu ti hned ukázat konkrétní návrh, jak by vypadal package.json + extension.ts pro command UXP: Attach, který spustí built-in JS debugger přes vscode.debug.startDebugging(...).

Hledám konkrétní open-source VS Code rozšíření, která neimplementují celý debugger, ale jen orchestruji attach/launch a pak spouští vestavěný debugger přes veřejné VS Code API. Zkusím najít rovnou i kódové místo, kde volají vscode.debug.startDebugging(...), aby to bylo použitelné jako vzor.

Thought for 3m 10s

Našel jsem několik velmi blízkých open-source příkladů. Úplně přesný „Adobe UXP debugger pro VS Code nad CDP“ jsem nenašel, ale tyhle projekty jsou architektonicky skoro přesně to, co jsem popisoval.

1) expo/vscode-expo — nejlepší reálný vzor pro „vlastní extension + reuse built-in js-debug“
Tohle je asi nejbližší praktický příklad. Extension si registruje vlastní debug provider a vlastní command, při spuštění debug commandu volá vscode.debug.startDebugging(...) se svým vlastním debug typem, a pak v resolveDebugConfigurationWithSubstitutedVariables explicitně přepne config.type = 'pwa-node' s komentářem „Reuse the pwa-node debug adapter from vscode.js-debug“. Zároveň dopočítává websocketAddress, localRoot, remoteRoot a další parametry pro CDP attach. To je přesně ten styl: tvé rozšíření řeší discovery/orchestration, ale samotný debug engine je js-debug.

2) microsoft/vscode-cdp-proxy — nejlepší vzor pro ten CDP shim/proxy
Tohle není hotová marketplace extension, ale sample repo od Microsoftu přímo pro scénář, kdy chceš rozšířit vscode-js-debug na nestandardní CDP targety. README doslova popisuje flow: tvoje extension spustí CDP proxy, ta poslouchá třeba na localhost:3000, a pak se na ni js-debug připojí jako na běžný pwa-chrome attach target. Microsoft tam výslovně zmiňuje „esoteric targets“ jako React Native nebo WebAssembly-based runtimes. To je velmi silná opora pro UXP variantu s MITM/proxy vrstvou.

3) mpotthoff/vscode-android-webview-debug — dobrý příklad „vlastní debug type nad js-debug attach mode“
README toho rozšíření přímo říká, že „uses the vscode-js-debug extension under the hood“ a že podporuje všechny konfigurační volby pwa-chrome: attach. Uživateli přitom nabízí vlastní debug typ android-webview, plus vlastní pole jako device/application/urlFilter. To je hodně podobné tomu, co bys chtěl pro UXP: uživatelsky hezký vlastní debug typ, ale pod kapotou reuse browserového/CDP debuggeru.

4) microsoft/vscode built-in extension debug-server-ready — čistý příklad orchestrace bez vlastního debuggeru
Tohle je vestavěná open-source extension ve zdrojácích VS Code. Když aplikace vypíše URL/port, extension programově spustí pwa-chrome nebo pwa-msedge přes vscode.debug.startDebugging(...). Je to hezký důkaz, že architektura „extension jen orchestrace, skutečný debugger je built-in js-debug“ je ve VS Code normální a podporovaný vzor, ne hack.

Za mě jsou pro tebe nejdůležitější tyto dva vzory:

Expo jako vzor pro vlastní extension/debug type, která nakonec deleguje na pwa-node/js-debug.
vscode-cdp-proxy jako vzor pro shim mezi nestandardním targetem a js-debug.

Pro Adobe UXP bych z toho složil architekturu takto:

UXP extension command + target discovery jako u Expo,
případný CDP shim/proxy jako ve vscode-cdp-proxy,
a finální attach do js-debug jako u Android WebView/Chrome attach modelu.

Jedna poctivá poznámka: nenašel jsem otevřený projekt, který by přesně dělal „Adobe UXP → VS Code → js-debug přes CDP shim“. Našel jsem ale velmi blízké a relevantní vzory, ze kterých se to dá poskládat bez velkého domýšlení. Adobe samo pro UXP dokumentuje debugging přes UXP Developer Tool / Chrome DevTools, takže CDP směr dává smysl, ale konkrétní open-source VS Code implementaci pro UXP jsem v dostupných výsledcích nenašel.

Repo odkazy, které bych otevřel jako první:

expo/vscode-expo – src/expoDebuggers.ts
microsoft/vscode-cdp-proxy – README + demo.js
mpotthoff/vscode-android-webview-debug – README a pak zdroj src/
microsoft/vscode – extensions/debug-server-ready/src/extension.ts

Můžu ti z těchto čtyř vzorů rovnou navrhnout konkrétní skeleton pro vscode-uxp-debug, včetně toho, co převzít z Expo, co z cdp-proxy a kde přesně volat startDebugging().
